## Получение изменений `git pull`
Команда `git pull` используется для получения и интеграции изменений из удалённого репозитория в вашу локальную рабочую копию. Это сочетание двух команд: `git fetch` (получение изменений) и `git merge` (слияние изменений с вашей текущей веткой).

### Синтаксис:

```bash
git pull [options] [remote_repository] [branch]
```
- `[remote_repository]` — это имя удалённого репозитория, например, `origin`.
- `[branch]` — это ветка, из которой вы хотите получить изменения (по умолчанию используется ветка, с которой у вас настроено отслеживание).

### Примеры использования:
1. Получение изменений с основной ветки:

```bash
git pull origin main
```
Получает последние изменения из ветки `main` удалённого репозитория `origin` и сливает их с вашей текущей веткой.

2. Получение изменений и автоматическое слияние:

```bash
git pull
```
Если у вас настроена отслеживаемая ветка (например, `origin/main`), эта команда выполнит `git fetch` для получения изменений и `git merge` для их слияния.

3. Получение изменений без автоматического слияния: Если вы хотите только получить изменения, но не сливать их сразу, используйте опцию `--no-commit`:

```bash
git pull --no-commit
```
Это позволяет вам сначала просмотреть изменения перед слиянием.

### Полезные опции:
- `--rebase` — вместо слияния изменений (`merge`) применяет ваши локальные изменения поверх обновлённой истории, что делает историю более линейной.

```bash
git pull --rebase origin main
```
- `--ff-only` — гарантирует, что слияние будет выполнено только в случае "быстрого вперёд" (fast-forward), иначе команда завершится с ошибкой.

```bash
git pull --ff-only origin main
```
- `--no-ff` — принудительно создаёт новый коммит слияния, даже если это можно было бы сделать с использованием "быстрого вперёд".

```bash
git pull --no-ff origin main
```

### Когда использовать git pull?
1. Когда вы хотите синхронизировать свою локальную копию с удалённым репозиторием.
2. Когда другие участники проекта внесли изменения, и вам нужно получить их и интегрировать с вашими локальными изменениями.
3. Когда вы хотите быть уверены, что ваша локальная копия проекта актуальна перед тем, как начать новые изменения.
